## 三、NestJS 路由设计与实现

### 3.1 路由规范与结构设计

![10-7创建控制器：定义用户模块并创建控制器(2)_img16.png](./images/10-7创建控制器：定义用户模块并创建控制器(2)_img16.png)

#### 3.1.1 路径命名规范

- **推荐风格**：小写加中横线（kebab-case），如 `get-users`
- **避免风格**：驼峰命名（camelCase）或下划线命名（snake_case）
- **示例对比**：
  - 推荐：`api/v1/users/list`
  - 不推荐：`api/v1/users/getList` 或 `api/v1/users/get_list`

#### 3.1.2 后端路由结构

- **标准格式**：`API/版本号/功能模块`
- **示例**：`api/v1/users`、`api/v1/posts`
- **优势**：清晰区分接口版本，便于维护和扩展

#### 3.1.3 请求类型与路径定义

| 请求方法 | 路径 | 功能 |
|---------|------|------|
| GET | `/users` | 获取用户列表 |
| GET | `/users/:id` | 获取单个用户 |
| POST | `/users` | 创建用户 |
| PUT | `/users/:id` | 更新用户 |
| DELETE | `/users/:id` | 删除用户 |

### 3.2 NestJS 控制器基础

![10-7创建控制器：定义用户模块并创建控制器(2)_img1.png](./images/10-7创建控制器：定义用户模块并创建控制器(2)_img1.png)

#### 3.2.1 控制器核心功能

- **路由解析**：接收前端请求，分发到对应的处理方法
- **请求适配**：根据 HTTP 方法（GET/POST/PUT/DELETE）区分不同处理逻辑
- **参数处理**：解析路径参数、查询参数和请求体

#### 3.2.2 路由参数传递

- **路径参数**：`@Param('id') id: string`
- **查询参数**：`@Query('page') page: number`
- **请求体**：`@Body() createUserDto: CreateUserDto`

**示例代码**：
```typescript
@Get(':id')
getUser(@Param('id') id: string) {
  return this.userService.findOne(id);
}
```

### 3.3 NestJS 注解式路由

![10-7创建控制器：定义用户模块并创建控制器(2)_img3.png](./images/10-7创建控制器：定义用户模块并创建控制器(2)_img3.png)

#### 3.3.1 注解优势

- **简洁性**：无需手动配置路由映射
- **可读性**：路由定义与处理方法紧密结合
- **类型安全**：TypeScript 类型检查

#### 3.3.2 常用路由注解

| 注解 | HTTP 方法 | 用途 |
|------|-----------|------|
| `@Get()` | GET | 查询资源 |
| `@Post()` | POST | 创建资源 |
| `@Put()` | PUT | 更新资源 |
| `@Delete()` | DELETE | 删除资源 |
| `@Patch()` | PATCH | 部分更新资源 |

### 3.4 路由前缀策略

![10-7创建控制器：定义用户模块并创建控制器(2)_img4.png](./images/10-7创建控制器：定义用户模块并创建控制器(2)_img4.png)

#### 3.4.1 方案一：全路径定义

```typescript
@Controller()
export class UserController {
  @Get('api/v1/users')
  getUsers() {}
  
  @Get('api/v1/users/:id')
  getUser() {}
}
```

**适用场景**：非功能性模块、跨模块路由

#### 3.4.2 方案二：控制器前缀

```typescript
@Controller('api/v1/users')
export class UserController {
  @Get()
  getUsers() {}
  
  @Get(':id')
  getUser() {}
}
```

**适用场景**：功能内聚的模块（如用户管理模块）

#### 3.4.3 方案三：全局前缀 + 控制器前缀

```typescript
// main.ts
app.setGlobalPrefix('api/v1');

// user.controller.ts
@Controller('users')
export class UserController {
  @Get()
  getUsers() {}
}
```

**最终路径**：`api/v1/users`

#### 4.1.1 注解本质

- TypeScript 注解本质是一个函数
- 接收字符串参数，返回一个装饰器函数
- 装饰器函数接收目标类作为参数

**简化示例**：
```typescript
function Controller(path: string) {
  return function (target: Function) {
    // 装饰器逻辑
  };
}
```

## 五、NestJS 核心概念与实践

### 5.1 封装与依赖注入

![10-7创建控制器：定义用户模块并创建控制器(2)_img8.png](./images/10-7创建控制器：定义用户模块并创建控制器(2)_img8.png)

#### 5.1.1 封装思想

- **关注点分离**：控制器负责路由解析，服务层负责业务逻辑
- **代码复用**：服务层逻辑可被多个控制器调用
- **可测试性**：便于单元测试和集成测试

#### 5.1.2 依赖注入

```typescript
@Controller('users')
export class UserController {
  constructor(private userService: UserService) {}
  
  @Get()
  getUsers() {
    return this.userService.findAll();
  }
}
```

### 5.2 完整请求路径构成

```
全局前缀 + 控制器前缀 + 方法路径 = 完整请求路径
```

**示例**：
- 全局前缀：`api/v1`
- 控制器前缀：`users`
- 方法路径：`:id`
- 最终路径：`api/v1/users/:id`

### 5.3 RESTful API 设计原则

1. **资源导向**：URL 表示资源，而非动作
2. **HTTP 方法语义**：使用合适的 HTTP 方法表示操作类型
3. **状态码规范**：使用标准 HTTP 状态码表示响应结果
4. **无状态设计**：每个请求包含所有必要信息
