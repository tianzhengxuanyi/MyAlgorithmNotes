# 数据库设计

本视频讲述了数据库设计的四个核心环节（需求分析、逻辑设计、数据库创建、维护与优化），重点解析了ER图建模方法、三大范式（第一范式强调原子性、第二范式消除部分依赖、第三范式消除传递依赖）的定义与应用，并通过用户系统（users）、用户档案（profile）、操作日志（logs）及用户角色关联（users_roles）等具体表格设计，演示了一对一、一对多、多对多关系的规范化建表实践。

## 数据库设计四环节概述

### 数据库设计的四个环节

- **需求分析**：明确业务目标、核心闭环及涉及的人、物、事。
- **逻辑设计**：分析人物事之间的关联关系。
- **数据库创建**：将逻辑模型转化为物理数据库结构并实现存储。
- **维护与优化**：应对业务复杂化与数据膨胀，包括表拆分、索引设置、分库分表等；扩展提及分布式数据库、数据仓库、分库分表等概念。

## 需求分析实例：通用业务系统

### 用户系统与权限系统拆解

- **用户系统**包含：用户基础信息、用户登录信息、用户角色信息。
- **权限系统**包含：角色、角色权限信息；权限对应菜单与接口权限。

## 逻辑设计：ER图建模方法

### ER图核心元素

- **方块**：代表实体（即后续需建表的对象）。
- **菱形**：代表实体间的关系。
- **连线上的数字**：表示基数（如"1"代表一个，"M"代表多个）；示例为"一个student对应多个course"。
- **椭圆形**：代表实体的属性（字段/列），如Lecture实体含字段ID（唯一）、lecture name、course ID（外键关联课程）。

### ER图工具支持

可借助Power Designer、Navicat Model等工具绘制ER图，并自动生成SQL建表语句。

## 表格设计原则与三大范式

### 设计原则

- 表格组件（主键）兼具去重、查询与索引功能。
- 按业务、功能、关联关系合理分表，避免冗余，提升结构清晰度。

### 第一范式（1NF）：原子性

- **定义**：所有字段必须是不可再分的原子值。
- **反例**：院系与"高级职称"合并为一字段，因"高级职称"含"教授""副教授"等子项，违反原子性。
- **解决方案**：
  1. 扁平化（增加professor_num、associate_professor_num字段）；
  2. 外键关联（设title_type_id，指向职称字典表，其中ID=1为教授、ID=2为副教授）。

### 第二范式（2NF）：消除部分依赖

- **定义**：在满足1NF基础上，所有非主键字段必须完全依赖于整个主键，不得存在部分依赖。
- **反例**：职工项目信息表含employee_id、name、title、project_id、project_name；name、title仅依赖employee_id，project_name仅依赖project_id，存在部分依赖。
- **解决方案**：拆分为职工信息表（含employee_id、name、title_id）、职称表（含title_id、title、level）、项目信息表（含project_id、project_name）。

### 第三范式（3NF）：消除传递依赖

- **定义**：在满足2NF基础上，非主键字段之间不得存在函数依赖（即传递依赖）。
- **反例**：学生表含student_id、name、college、college_phone；college_phone不直接依赖student_id，而是通过college间接依赖，形成student_id → college → college_phone传递依赖，导致数据冗余与更新异常。
- **解决方案**：拆分为学生表（含student_id、name、college_id）与学院表（含college_id、college_name、college_phone），通过外键college_id关联。

## 实际建表实践：用户系统关系建模

### 一对一关系：users 与 profile

- **users表**：主键id（PK），字段含username（或email）、password。
- **profile表**：主键id（PK），字段含gender、photo、address，并设外键user_id（唯一）关联users.id。

### 一对多关系：users 与 logs

- **logs表**：主键id（PK），字段含user_id（外键，非唯一）、path、method、request_data、response_data、status_code；一个user_id可对应多条日志记录。

### 多对多关系：users 与 roles

- **roles表**：主键id（PK），字段含role_name。
- **关联表users_roles**：含user_id（外键）、role_id（外键），无独立主键；user_id与role_id共同构成复合主键或添加代理主键。

## 后续流程：从逻辑设计到物理创建

### 逻辑设计完成标志

ER图及规范化表格关系已明确。

### 物理数据库创建

将逻辑模型（如users、profile、logs、roles、users_roles等表结构）在真实数据库中执行SQL建表语句，完成部署。

## SQL建表语句示例

```sql
-- users表
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(50) UNIQUE NOT NULL,
  password VARCHAR(100) NOT NULL
);

-- profile表
CREATE TABLE profile (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT UNIQUE NOT NULL,
  gender VARCHAR(10),
  photo VARCHAR(255),
  address VARCHAR(255),
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- logs表
CREATE TABLE logs (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  path VARCHAR(255),
  method VARCHAR(10),
  request_data TEXT,
  response_data TEXT,
  status_code INT,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- roles表
CREATE TABLE roles (
  id INT PRIMARY KEY AUTO_INCREMENT,
  role_name VARCHAR(50) UNIQUE NOT NULL
);

-- users_roles关联表
CREATE TABLE users_roles (
  user_id INT NOT NULL,
  role_id INT NOT NULL,
  PRIMARY KEY (user_id, role_id),
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (role_id) REFERENCES roles(id)
);
```
