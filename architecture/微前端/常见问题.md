# 一、基础核心概念（必背）

## 什么是模块联邦？解决什么核心问题

- 跨工程共享代码、组件、工具库，无需打包进主包；运行时动态加载远程产物
- **解决**：多仓库代码重复打包、公共库版本不统一、子应用独立发版、微前端资源冗余

## 模块联邦和普通 npm 包的区别

- **npm**：编译时引入，打包合并；更新依赖需要主工程重新打包发布
- **模块联邦**：运行时动态拉取远端资源，子应用单独部署，Host 无需重发

## webpack 联邦 vs vite-plugin-federation 差异

- **webpack 联邦**：成熟、生态全；构建慢、不兼容 ESM
- **vite 联邦**：原生 ESM、构建速度快、适配 Vue3 工程；部分复杂场景兼容性差

## 模块联邦能否跨技术栈？

可以，但成本高；Vue Host 加载 React Remote 需要额外适配渲染层、共享运行时；企业内部统一技术栈最优

# 二、三大核心角色：Host、Remote、Shared

## Host（基座主应用）

- **作用**：页面入口、提供全局路由、登录鉴权、共享基础依赖、承载页面布局
- 可同时作为 Host，也可以暴露自身模块充当 Remote

## Remote（远程子应用 / 远程组件库）

- 独立 Vite 工程，打包产出 remoteEntry 入口文件
- Host 通过 url 动态拉取 remoteEntry，注册后使用其暴露组件 / 方法

## shared 共享依赖（重中之重）

### shared 作用

多应用共用同一套第三方库（vue、pinia、element-plus），避免重复打包、多实例冲突

### shared 三种配置策略

1. **strictVersion**：严格匹配版本，版本不一致报错告警
2. **singleton: true**：单例模式，全局仅保留一份实例（解决 Vue 多实例）
3. **requiredVersion**：指定兼容版本区间

### 共享优先级

Host 加载优先，Remote 复用 Host 已加载依赖，不会二次请求

**federation.shared/import 作用**
shared + 内部的 import 字段，只控制运行时依赖共享逻辑：
- import: false：远程模块优先尝试复用 Host 已加载的 shared 依赖；如果找不到再自己加载一份；
- import: true：远程模块主动先加载自身的依赖副本，再判断是否复用基座实例；
它只影响浏览器运行时模块加载顺序、共享复用策略，不干预打包阶段是否把库打进 remoteEntry。

### 哪些必须放入 shared

vue、vue-router、pinia、UI 组件库、axios 等基础库

### 哪些不适合 shared

业务专属工具、页面业务组件、大体积独立插件

# 三、基础配置与使用流程（手写配置考点）

## Remote 侧配置：exposes 暴露组件 / 页面

- **exposes 配置格式**：路径：对外导出名称
- 打包产出 remoteEntry.js，作为远程唯一入口

## Host 侧配置 remotes 注册远程模块

- **静态注册**：写死远程 CDN 地址
- **动态注册**：运行时接口获取远程地址，动态添加 remote，实现动态菜单、动态业务面板

## 远程组件两种引入方式

1. **静态导入**：`import RemoteComp from 'remote_name/xxx'`
2. **动态异步导入**：defineAsyncComponent + Suspense 做加载兜底

## 开发环境 / 生产环境配置区别

- **开发**：本地localhost代理远程工程，热更新互通
- **生产**：CDN 静态资源地址，缓存策略配置

## TS 类型共享方案

Remote 打包导出 d.ts，Host 通过 http 引入远端类型；或公共类型库抽离成 npm 包统一维护

# 四、样式隔离全套知识点（高频深挖）

模块联邦无 iframe 天然隔离，同全局 CSS 上下文，必须手动隔离，全套方案：

- `<style scoped>` 组件级隔离原理、data-v 唯一标识
- 业务模块全局命名空间隔离（每个 Remote 外层统一 class 前缀 remote-xxx）
- CSS Modules 哈希类名隔离，适合公共组件库
- 禁止全局无嵌套选择器：`*`、`div`、`.btn` 通用类污染
- `:deep()` 穿透样式风险，禁止全局穿透修改第三方组件

## UI 库冲突解决方案

- Host singleton 共享一套 UI 库，Remote 不重复引入样式
- 修改 UI 组件前缀，弹窗挂载至当前模块根节点而非 body
- 加载顺序问题：后加载 Remote 全局样式覆盖先加载模块，命名空间是唯一兜底方案

# 五、JS 全局隔离、跨应用通信

## 全局 window 变量污染如何解决

- 禁止直接挂载变量到 window，全部放入 pinia 全局状态
- 各 Remote 业务变量增加命名空间前缀

## 跨模块通信三种方案

1. **Pinia 全局状态库**（推荐，Host 唯一 store 实例，shared 共享）
2. 全局自定义事件总线 EventEmitter
3. **父传子**：Host 通过 props 向远程业务页面下发鉴权、权限数据

## Vue 多实例问题根源与解决

- **根源**：Remote 自带 vue 包，未配置 shared singleton
- **解决**：shared 配置 vue singleton:true，Host 优先加载 Vue

# 六、你项目特有：模块联邦替换 Wujie 全套考点（核心简历亮点）

## 原有 wujie 方案存在什么致命问题

- 底层基于 iframe + 沙箱隔离
- 多 Tab 页面搭配 keep-alive 缓存时，iframe、沙箱上下文、监听事件无法完全销毁
- 多标签叠加内存持续暴涨，页面卡顿、浏览器内存溢出崩溃

## 为什么模块联邦能解决内存泄漏

- 无 iframe、无独立沙箱，共用一套页面上下文
- 完全原生 Vue 组件生命周期，keep-alive 缓存 / 销毁逻辑可控，组件销毁正常 GC 回收
- shared 单例基础库，无重复实例占用内存

## wujie 和 vite 模块联邦优缺点对比

| 维度 | wujie (iframe 微前端) | vite 模块联邦 |
|------|----------------------|---------------|
| 隔离 | 天然 JS + 样式隔离 | 无隔离，手动实现 |
| 内存 | keep-alive 场景易泄漏 | 内存开销低，无 iframe 残留 |
| 通信 | iframe 跨上下文通信成本高 | 同上下文，状态 / 事件通信简单 |
| 技术栈 | 支持跨栈 | 统一栈体验最佳，跨栈适配复杂 |
| 打包 | 子应用完整打包全套依赖 | shared 抽离公共依赖，包体积更小 |

## 替换改造踩坑重点

- Vue 多实例冲突
- 全局弹窗挂载 body 样式污染
- 远程模块加载异常兜底处理
- 多业务模块路由统一管理、鉴权同步

# 七、异常处理、降级、动态加载

## 远程资源 404、网络超时、加载失败兜底方案

- 异步捕获 import 错误
- 统一封装远程组件高阶组件，展示 loading、错误兜底页面
- 设置请求超时时间、失败重试机制

## 动态注册 remote 实现动态菜单

- 接口获取所有远程模块配置，运行时动态添加 remotes
- 路由懒加载对应远程页面
- 预加载优化：进入首页提前预加载常用远程模块，减少切换等待

---

# 八、路由、鉴权、权限统一管理（业务必问）

- 基座统一路由管理，Remote 不独立创建路由实例
- 全局登录 token、无感刷新逻辑统一在 Host 实现，远程模块复用 axios 实例（shared）
- **RBAC 权限下发**：Host 获取用户角色、按钮权限，通过状态库 / Props 下发各 Remote
- 全局请求拦截器、响应拦截器统一封装，所有远程组件共用

---

# 九、性能优化知识点

- shared 抽离公共依赖，减小 Remote 打包体积，减少网络请求
- 远程组件异步懒加载 + Suspense Loading
- CDN 缓存 remoteEntry、业务产物，设置长效缓存策略
- 非首屏远程模块预加载 / 按需加载
- 避免多个 Remote 重复引入大型第三方库
- 长列表、低代码画布使用虚拟滚动降低 DOM 节点，减少重排重绘

---

# 十、工程化、部署、CI/CD

- 各 Remote 独立 Vite 工程，独立打包、独立部署静态资源
- Host 基座单独部署，无需和子应用同步发版；更新业务模块不用重新发布基座
- 打包产物区分开发 / 生产环境地址
- **监控统一**：Host 全局捕获所有 Remote 报错，统一上报监控系统
- **版本管理**：Remote 迭代版本号，Host 可灰度切换远程资源地址
